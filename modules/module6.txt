1. Comparison Operators (>, <, >=, <=)
Theory
Comparison operators compare two values and return a strict Boolean (true or false). They evaluate if the left value is greater than (>), less than (<), greater than or equal to (>=), or less than or equal to (<=) the right value. If you compare two strings, JavaScript evaluates them alphabetically, character by character.

The Misconception Trap
Analyze this code. Why does the system identify 2 as the highest number?

JavaScript
let currentLevel = "10";
let nextLevel = "2";

if (currentLevel > nextLevel) {
    console.log("Proceeding to level 10.");
} else {
    console.log("Error: Level scaling failed.");
}
The Flaw: The output logs "Error: Level scaling failed." because the code is comparing strings, not numbers. When evaluating "10" > "2", the engine checks the very first character of both strings. Because "1" is alphabetically smaller than "2", the expression immediately evaluates to false.

2. Loose Equality (== and !=)
Theory
Loose equality (==) checks if two values are equal, but it completely ignores their data type. If the types do not match, the JavaScript engine performs background coercion, changing the types automatically to force a comparison. != checks if values are loosely not equal. Using these operators is dangerous and strongly discouraged because they cause unpredictable bugs.

The Misconception Trap
Analyze this code. Why does the system clear the cache even when the data array is completely empty?

JavaScript
let cachedData = [];
let targetState = 0;

if (cachedData == targetState) {
    console.log("Memory match. Clearing cache.");
}
The Flaw: The loose equality operator attempts to force an empty array [] into a number to compare it with 0. During background coercion, an empty array converts to an empty string "", and an empty string converts to the number 0. Therefore, 0 == 0 evaluates to true, and the system accidentally wipes the cache.

3. Strict Equality (=== and !==)
Theory
Strict equality (===) compares both the value AND the data type. No background coercion is allowed. If the data types are different, the engine instantly stops and returns false. Strict inequality (!==) returns true only if the value or the type are completely different. This is the absolute standard for professional JavaScript.

The Misconception Trap
Analyze this code. Why does the system fail to recognize the missing data error?

JavaScript
let systemError = NaN;

if (systemError === NaN) {
    console.log("Critical Failure Detected.");
} else {
    console.log("System Stable.");
}
The Flaw: It logs "System Stable." even though a critical failure occurred. In JavaScript, NaN (Not-A-Number) is a unique mathematical defect. It is the only value in the entire language that is strictly not equal to itself. NaN === NaN will always return false. (You must use Number.isNaN() to catch it).

4. The if statement block
Theory
The if statement acts as a gatekeeper. It evaluates the exact condition placed inside its parenthesis (). If the condition evaluates to any Truthy value, the engine opens the gate and executes the code block contained inside the curly braces {}. If the condition is Falsy, the engine skips the block entirely.

The Misconception Trap
Analyze this code. Why does the restricted action execute even when the isAdmin token evaluates to false?

JavaScript
let isAdmin = false;

if (isAdmin === true); {
    console.log("Deleting Database...");
}
The Flaw: The developer accidentally placed a semicolon ; directly after the if parenthesis. A semicolon represents the end of an instruction. The engine reads the if statement, evaluates it as false, and finishes the instruction at the semicolon doing nothing. The block {} beneath it is now treated as completely standalone, standard code, and executes immediately, deleting the database.

5. The else block
Theory
The else block acts as an automatic fallback. It is directly chained to the end of an if block. It does not accept a condition in parenthesis (). It only executes if the preceding if condition evaluates to false. It guarantees that the system has a safe default action to take when primary checks fail.

The Misconception Trap
Analyze this code. Why does it throw a fatal Syntax Error before it even runs?

JavaScript
let batteryLevel = 10;

if (batteryLevel > 20) {
    console.log("Device running normally.");
} 
console.log("Checking diagnostics...");
else {
    console.log("Please plug in the charger.");
}
The Flaw: An else block must be chained immediately following the closing curly brace } of an if block. In this code, the developer placed a standalone console.log() between the if and the else. This instantly breaks the chain, causing the compiler to crash because it encounters an orphaned else without a preceding if.

6. The else if block for multiple conditions
Theory
The else if block allows you to check multiple specific conditions sequentially. You place it between the initial if and the final else. The JavaScript engine evaluates these conditions strictly from top to bottom. It stops executing at the very first condition that is true, runs that block, and completely ignores the rest of the chain.

The Misconception Trap
Analyze this code. Why does a student with a score of 95 receive a basic "B" grade instead of an "A"?

JavaScript
let examScore = 95;

if (examScore > 50) {
    console.log("Grade: C");
} else if (examScore > 75) {
    console.log("Grade: B");
} else if (examScore > 90) {
    console.log("Grade: A");
}
The Flaw: The system evaluates top-to-bottom. The first check asks: "Is 95 > 50?". This evaluates to true. The system logs "Grade: C" and immediately terminates the entire chain. To fix this logic flaw, you must evaluate the most restrictive conditions (highest numbers) at the top of the chain.

7. Logical Operators (&& AND, || OR, ! NOT)
Theory
Logical operators combine multiple conditions into a single check.

&& (AND): Every single condition must be true. If just one is false, the entire expression evaluates to false.

|| (OR): Only one condition needs to be true. If the first condition is true, it stops checking and passes.

! (NOT): Flips a boolean. True becomes false, and false becomes true.

The Misconception Trap
Analyze this code. A user inputs an empty password "". Why does the validation system authorize them?

JavaScript
let inputPassword = "";
let overrideCode = "ADMIN_77";

if (!inputPassword || overrideCode) {
    console.log("Authorization Successful.");
}
The Flaw: This is a short-circuit precedence trap. The engine checks !inputPassword first. "" is falsy, so ! flips it to true. Because the engine found a true statement on the left side of an OR (||) operator, it short-circuits. It stops reading the rest of the line and immediately runs the block, authorizing a user with a blank password.

8. The Ternary Operator (condition ? true : false)
Theory
The ternary operator is a fast, one-line shortcut for simple if/else assignments. It consists of three parts: the condition to check, followed by a question mark ?, the value to return if true, followed by a colon :, and the value to return if false. It must return a value; it should not be used to execute large blocks of actions.

The Misconception Trap
Analyze this code. Why does the compiler crash with an assignment error?

JavaScript
let playerPoints = 150;
let rank;

playerPoints > 100 ? let rank = "Pro" : let rank = "Amateur";
The Flaw: You cannot declare variables or write complete statements inside the return blocks of a ternary operator. The ternary operator evaluates to a single value. The entire operator must be assigned to the variable. The correct syntax is: let rank = playerPoints > 100 ? "Pro" : "Amateur";.
