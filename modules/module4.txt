#### 1. Basic Arithmetic Operators (`+`, `-`, `*`, `/`)

**Theory**
JavaScript uses standard mathematical symbols to perform calculations. Addition (`+`), subtraction (`-`), multiplication (`*`), and division (`/`) work exactly as they do in a calculator. However, if you use the `+` operator with a string, it prioritizes text concatenation over mathematical addition.

**The Misconception Trap**
*Analyze this code. What is the exact mathematical output stored in `finalValue`?*

```javascript
let finalValue = "10" + 5 - 2;

```

*The Flaw:* The output is the integer `103`, not `13`. JavaScript evaluates from left to right. First, `"10" + 5` uses the `+` operator, which triggers string concatenation, resulting in the text `"105"`. Then, the engine encounters `- 2`. Since you cannot subtract text, JavaScript's engine steps in, implicitly forces the text `"105"` back into a number `105`, and subtracts `2`.

#### 2. The Modulo Operator (`%`)

**Theory**
The modulo operator (`%`) does not divide numbers. Instead, it divides two numbers and returns *only the remainder*. If a number divides perfectly, the modulo is `0`. It is heavily used in programming to determine if a number is even (`number % 2 === 0`) or odd (`number % 2 === 1`).

**The Misconception Trap**
*Analyze this code. Why does the system incorrectly log the alert?*

```javascript
let totalItems = 3;
let containerCapacity = 10;
let remainder = totalItems % containerCapacity;

if (remainder === 0) {
    console.log("All items packed perfectly.");
}

```

*The Flaw:* The logic assumes `3 % 10` will evaluate to `0` or crash because you cannot divide a smaller number by a larger number. In reality, if the first number is smaller than the second, the modulo operator simply returns the first number entirely. `3 % 10` equals `3`. The `if` statement evaluates `3 === 0`, which is false, skipping the block.

**Hidden Test Cases for Code Editor**
*Mission:* Use the modulo operator to find the remainder of `15` divided by `4`, and store it in a variable. Then, calculate the modulo of `2` divided by `7`.

* **Test Case 3:** Memory validation checks that the second remainder variable strictly equals `2`.
* **Test Case 4:** Fail condition if the division operator `/` is used instead of `%`.
* **Test Case 5:** Verify the output type is strictly a `"number"`.

#### 3. Assignment Operators (`=`, `+=`, `-=`, `*=`, `/=`)

**Theory**
Assignment operators are shortcuts for modifying a variable's existing memory. Instead of writing the long-form `score = score + 5`, you can use the shorthand `score += 5`. This updates the variable in place. This works for all basic math operations (`-=`, `*=`, `/=`).

**The Misconception Trap**
*Analyze this code. Why does the player's health instantly drop to 20 instead of decreasing by 20?*

```javascript
let playerHealth = 100;
playerHealth =- 20; 

```

*The Flaw:* The developer accidentally flipped the assignment operator. They wrote `=-` instead of `-=`. The engine does not read this as "subtract 20". It reads it as `playerHealth = (-20)`. The variable is simply reassigned to a negative twenty.

**Hidden Test Cases for Code Editor**

* **Test Case 4:** AST validation confirms the exact presence of the `*=` operator.

#### 4. Increment and Decrement Operators (`++`, `--`)

**Theory**
These are micro-shortcuts used to add exactly `1` or subtract exactly `1` from a variable. Writing `x++` is the same as `x += 1`. They are primarily used in loops to keep track of counts.

**The Misconception Trap**
*Analyze this code. What are the final values of `a` and `b`?*

```javascript
let a = 5;
let b = a++;

```

*The Flaw:* A beginner assumes both `a` and `b` become `6`. This is the trap of "Postfix" vs "Prefix" incrementing. Because the `++` is written *after* `a` (`a++`), JavaScript assigns the original value of `a` (which is `5`) to `b` first, and *then* increments `a` in the background. The result: `b` is `5`, and `a` is `6`. (To make both 6, it must be written as prefix: `let b = ++a;`).

**Hidden Test Cases for Code Editor**
*Mission:* Declare a `counter` at `0`. Use the postfix increment operator to increase it. Then use the prefix decrement operator to decrease it.

* **Test Case 1:** AST confirms `counter` is initialized with `let` at `0`.
* **Test Case 2:** AST Parser verifies the use of an `UpdateExpression` with the `++` operator in postfix position (`counter++`).
* **Test Case 3:** AST Parser verifies the use of an `UpdateExpression` with the `--` operator in prefix position (`--counter`).
* **Test Case 4:** Memory state confirms `counter` successfully returns to `0`.
* **Test Case 5:** System restricts the use of `+= 1` or `-= 1` to force the use of `++` and `--`.

#### 5. Operator Precedence (BODMAS/PEMDAS in JavaScript)

**Theory**
JavaScript executes math following standard algebraic rules (PEMDAS: Parentheses, Exponents, Multiplication/Division, Addition/Subtraction). Multiplication and division are calculated *before* addition and subtraction, regardless of how they are spaced visually. To override this default order, wrap the priority calculation in parentheses `()`.

**The Misconception Trap**
*Analyze this code. Calculate the exact numeric output.*

```javascript
let complexEquation = 10 - 2 * 3 + 4 / 2;

```

*The Flaw:* Reading strictly left-to-right yields `14` (10-2=8, 8*3=24, 24+4=28, 28/2=14). This is completely wrong. Following strict operator precedence: First, `2 * 3 = 6`. Next, `4 / 2 = 2`. The equation is rewritten internally as `10 - 6 + 2`. Addition and subtraction hold equal weight, so they evaluate left-to-right. `10 - 6 = 4`. Finally, `4 + 2 = 6`. The output is `6`.

**Hidden Test Cases for Code Editor**
*Mission:* Calculate the average of three test scores: `80`, `90`, and `100`. Store the result in an `average` variable. You must use parentheses.

* **Test Case 1:** AST validation enforces the presence of parentheses `()` surrounding the addition operations.
* **Test Case 2:** Memory validation confirms `average` equals exactly `90`.
* **Test Case 3:** Fail condition: If the AST does not detect parentheses, and the code looks like `80 + 90 + 100 / 3` (which evaluates to 203.33), reject the code.
* **Test Case 4:** Verify that division `/` happens *after* the grouping operation.

#### 6. The `Math` Object basics (`round`, `floor`, `ceil`, `random`)

**Theory**
JavaScript provides a built-in `Math` toolbelt for advanced calculations. `Math.round()` rounds to the nearest integer. `Math.floor()` forces the decimal down to the lower integer. `Math.ceil()` forces the decimal up to the higher integer. `Math.random()` generates a random decimal between `0` (inclusive) and `0.9999...` (exclusive).

**The Misconception Trap**
*Analyze this code. What does it evaluate to?*

```javascript
let mappedValue = Math.floor(-2.5);

```

*The Flaw:* A developer usually thinks `Math.floor()` just chops off the decimal, assuming the answer is `-2`. However, `floor` forces the value to the *lowest possible mathematical integer*. On a number line, `-3` is lower than `-2.5`. Therefore, `Math.floor(-2.5)` actually rounds down to `-3`.

**Hidden Test Cases for Code Editor**
*Mission:* Create three variables. `var1` should round `4.7` naturally. `var2` should force `4.1` to the ceiling. `var3` should chop `4.9` to the floor.

* **Test Case 1:** AST Parser strictly verifies the method call `Math.round(4.7)`.
* **Test Case 2:** AST Parser strictly verifies the method call `Math.ceil(4.1)`.
* **Test Case 3:** AST Parser strictly verifies the method call `Math.floor(4.9)`.
* **Test Case 4:** Memory state confirms `var1` equals `5`.
* **Test Case 5:** Memory state confirms `var2` equals `5`.
* **Test Case 6:** Memory state confirms `var3` equals `4`.
* **Test Case 7:** (For `random` checks): System mocks the `Math.random` function in the background to return exactly `0.5`, then validates if the user's implementation scales it properly if a challenge requires a specific range.

**The Scenario**
You are the lead backend developer for a highly competitive AI-driven study application. The platform uses an API to generate study flashcards and ranks students on a global leaderboard.

Top-tier students are getting incorrect scores due to math precedence failures, token billing is rounding decimals incorrectly, and the server routing algorithm is using division instead of remainders. // 1. Leaderboard Math

// 2. API Token Deduction

**The Mission Objectives (Strict Requirements)**
You must refactor the mathematical logic to satisfy the system's strict parameters. Fix the following issues:

1. **Operator Precedence (PEMDAS):** The `finalScore` is currently multiplying `20 * 1.5` first, resulting in `530`. You must use parentheses to ensure the `baseScore` and `streakBonus` are added together *before* being multiplied by the multiplier.

Change this to force the decimal *down* to the lower integer regardless of the decimal size. Additionally, rewrite the token deduction line using the shorthand assignment operator (`-=`).
3. **Increment Operators:** Remove the clunky `loginDays + 1` logic. Directly update the `loginDays` variable using the JavaScript increment operator (`++`).
4. **The Modulo Operator:** The `serverShard` is performing standard division, resulting in decimals. The server routing requires the *remainder* of the `finalScore` divided by `3`. Replace the division operator with the correct modulo operator.
5. **Random Integer Generation:** `challengeId` is currently generating a long, useless decimal (like `4.8273`). Wrap the existing mathematical equation inside a `Math` method that chops off the decimal, generating a clean, whole integer between `0` and `9`.
